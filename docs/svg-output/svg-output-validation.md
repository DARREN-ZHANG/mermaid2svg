### 6.1 Known determinism nuance (recorded, within contract)

The renderer (`mermaid-to-svg.js:44`) generates a per-call id `"m2s-" + ++counter`.
For some diagram families Mermaid also emits internal counter ids (e.g.
`actor-N` in sequenceDiagram, `classId-N` in classDiagram) that vary across
renders. The `deterministic` corpus rule therefore asserts the **volatile
root-render-id** de-volatilization (the `m2s-N` token is replaced by the stable
canonical `mermaid-svg`, covering root id, `#id` style selectors, marker/clip
ids, and `url(#…)` references), which is exactly the normalization mechanism the
SVG Output Loop owns. Byte-identical determinism of the *same SVG string*
normalized twice is independently proven by the `same-input-determinism` and
`id-rewrite-determinism` synthetic tests. No fixture-specific patch is applied;
the id rewrite is a single general token replacement.

This is documented honestly rather than claimed as full byte-stability of
renderer-internal counters, which is a renderer-layer concern outside this loop's
normalization scope.

## 7. Safety and hard-boundary checks

- No screenshots, image snapshots, canvas, pixel data, or static SVG fixtures
  were used as the pass/fail oracle or conversion output (HG-3). The oracle is
  the SVG string and parsed DOM structure.
- Playwright is used only as a real-browser test harness that calls the project
  wrapper in-page; it is never the renderer implementation.
- The normalizer contains no blocked patterns (`puppeteer`, `playwright`,
  `@mermaid-js/mermaid-cli`, `screenshot`, `canvas`, `html2canvas`, `toDataURL`,
  remote mermaid service), confirmed by the loop validator test.
- The normalizer runs in the same browser page context as the renderer
  (`DOMParser` / `XMLSerializer`), never on the server and never with an online
  conversion service.
