## 7. Risks / Unknowns

1. **Package identity**: Package is still named `@webc.site/math`. Renaming may be needed for Mermaid project identity, or it can stay as a monorepo that serves both purposes.

2. **Existing extract/ and test/ are MathML-focused**: Current `extract/run.js` extracts from KaTeX/MathJax. It must be rewritten for Mermaid. Current `test/case/*.yml` contains MathML comparison tests. Per spec, new Mermaid YAML tests go into `test/*.yml` (not `test/case/`).

3. **i18n system coupling**: The i18n system has 75 complete language files in `demo/i18n/`. The Mermaid demo needs new translation keys added to all 75 files. The structure is flat JS modules exporting functions returning translation objects. Adding keys to 75 files is labor-intensive.

4. **demo/index.pug is Math-specific**: The page title, subtitle, editor, formula gallery are all math-focused. This needs significant modification to become a Mermaid demo while preserving the visual design system.

5. **No Mermaid dependency yet**: The project currently has zero Mermaid-related dependencies. `mermaid` npm package must be added for browser-side rendering.

6. **Size/speed SVGs are static**: Current `demo/size.svg` and `demo/speed.svg` are pre-generated static SVGs. The spec requires a script-generated SVG bar chart comparing JS bundle sizes with `workflow/reports/size-report.json` as the data source.

7. **Web Components**: The `demo/webc/` directory contains reusable web components (I18n, Math, Box, Scroll, etc.). The Math component specifically uses `lib/mathml.js`. A new Mermaid component will be needed, while preserving the existing component infrastructure.

8. **Plugin directory**: `plugin/` contains markdown-it/marked/remark integrations for math rendering. These are unrelated to Mermaid but are part of the upstream project's value. Should be preserved as out-of-scope.

9. **Blog content**: `blog/` contains dev.to articles about the math project. Completely unrelated to Mermaid.

10. **Vite config**: Custom Vite plugins for Pug compilation, Stylus processing, and SVG inlining. These are infrastructure that should be preserved.

11. **Formatter**: `bun x oxfmt` in test.sh — any code generation must pass oxfmt style.

12. **Husky pre-commit hooks**: lint-staged runs SVG and Stylus processing hooks. Must be preserved.

13. **Workflow state**: `workflow/state/init-loop.state.json` shows the init loop is currently running, at "project-cognition" phase. This is expected — we are that phase.

14. **test.sh requires `bun test`**: Tests use Bun's built-in test runner (`bun:test`). New Mermaid tests should follow this pattern.

15. **minify.js measures gzip**: The existing `minify.js` already computes raw and gzip sizes per module and prints a table. This can be reused or extended for the size-report.json generation.

16. **demo/build.js copies i18n**: The build script copies `webc/I18n/i18n/` and `webc/BoxX/i18n/` to dist. If new i18n directories are added to webc components, build.js may need updating.
