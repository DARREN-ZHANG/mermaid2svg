# Size Loop Verification Report

> Phase: `verification`. Confirms that `workflow/reports/size-report.json`
> contains data generated from real on-disk artifacts, not estimates.

## 1. Mission

Verify that the four size numbers in the canonical report are produced by
measuring real files, not by hand-writing or estimating values.

## 2. Canonical report under test

`workflow/reports/size-report.json` — four numbers:

| Side | Metric | Value |
| --- | --- | --- |
| beautiful-mermaid | raw | 328094 |
| beautiful-mermaid | gzip | 66816 |
| ours | raw | 127082 |
| ours | gzip | 41785 |

## 3. Verification method

Each number was independently reproduced from the real file it claims to
measure, using a separate command, and the result was compared to the report.

### 3.1 Our entry chunk (real build artifact)

The report claims `ours.entry = demo/dist/assets/index-BzHJhuCY.js`.

| Check | Command | Result | Report | Match |
| --- | --- | --- | --- | --- |
| File exists | `ls demo/dist/assets/index-BzHJhuCY.js` | exists | — | ✅ |
| Raw bytes | `wc -c demo/dist/assets/index-BzHJhuCY.js` | 127082 | 127082 | ✅ |
| Gzip bytes | `bun -e "gzipSync(...).length"` | 41785 | 41785 | ✅ |
| sha256 | `shasum -a 256 ...` | b5a27279... | b5a27279... | ✅ |

**Conclusion: our numbers come from the real built entry chunk.**

### 3.3 Page data consistency

`demo/const/sizeData.js` is generated by the same script from the same
in-memory measurement. Verified all four values match the report byte-for-byte.

**Conclusion: page chart data == report data (AC-COMPARE-004).**

### 3.4 Reproducibility

`bun sh/size-report.js` was re-run and produced identical numbers on both
sides. The beautiful-mermaid bundle is deterministic when reproduced via the
same `execSync('bun build ...')` invocation path.

## 4. Findings

### 4.1 Gzip level (acceptable)

- **Report method:** `zlib.gzipSync(raw)` — default level (6).
- **Plan recommendation** (`docs/size/size-plan.md` §4.1): level 9.
- **Impact:** Both sides use the identical gzip method, so the comparison is
  internally fair. Level-9 reference values for transparency:

| Side | gzip level 6 (used) | gzip level 9 (reference) |
| --- | --- | --- |
| beautiful-mermaid | 66816 | 66021 |
| ours | 41785 | 41341 |

- **Action:** No change required for verification pass. A future size-loop
  iteration may switch to level 9 if desired; this does not affect the
  conclusion that data is real.

### 4.2 Aggregation scope (noted, deferred)

- **Report method:** `ours.rawBytes` measures only the single entry chunk
  (`index-HASH.js`).
- **Plan recommendation** (`docs/size/size-plan.md` §3.2): aggregate all
  reachable render-path JS chunks (entry + mermaid lazy chunks + layout libs).
- **Impact:** The single-chunk number is real (not estimated), but it
  understates the full client JS footprint compared to beautiful-mermaid's
  all-in-one bundle. This is a methodology gap, not a data-fabrication issue.
- **Action:** Deferred to a future size-loop iteration per HG-7 spirit (do not
  expand scope mid-verification). Recorded for transparency.

### 4.3 Blocked patterns check

| Pattern (from size-loop config) | Present in report/chart? |
| --- | --- |
| `runtime benchmark` | No — report says "performance proxy, not a runtime benchmark" |
| `estimated size` | No — all numbers verified from real files |
| `manual size data` | No — script-generated |

## 5. Verdict

**PASS.** All four size numbers are generated from real on-disk artifacts:

1. Our entry chunk exists in `demo/dist/`, raw and gzip bytes match.
2. beautiful-mermaid bundle is reproducible from the pinned local source at
   commit `2ac8bb`, raw and gzip bytes match.
3. Page chart data is consistent with the report.
4. No estimated or manual numbers detected.

Two methodology notes (gzip level, aggregation scope) are recorded for future
improvement but do not invalidate the current verification.
